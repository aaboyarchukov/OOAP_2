### Категории наследований
---
7. **Наследование реализации** (классическое наследование, когда потомку передается программная реализация его предка, отличия от расширения и спецификации в том, что в данной категории наследуется именно реаллизация, без выстраивания иерархии основываясь на предметной области | *расширение и специализация больше про домен*)
8. **Льготное наследование** (класс который содержит в себе объединенные константы/компоненты для использования в потомках при специализации)

### Наследование с реализацией
---
В курсе мы рассматриваем АТД как класс с интерфейсом и вспомогательными методами и полями для реализации. Проблема состоит в том, что при наследовании мы получаем не только чистый интерфейс, но и частичную реализацию, в таком случае при наследовании важно понимать:
- как именно организован доступ к закрытм полям и методам в конкретном ЯП
- насколько при наследовании реализация родителя будет защищена от изменений
Это важно понимать, так как мы формализуем только наш интерфейс, а реализацию - нет.
Если реализация выполнения на хорошем уровне и отлично встраивается в иерархию, тогда ее можно наследовать.

Другой выход без наследования реализации - плоские классы (имеют только интерфейс), но тогда придется писать повторяющийся код.
То есть в иерархии мы наследуем только интерфейс, когда как поля и методы у нас всегда закрыты для молификаций.

**1 схема:**
	*наследуем реализацию*
**2 схема:**
	*делаем плоский класс*

И сначала лучше отталкиваться от 2 схемы, но не пренебрегать наследованием, если реализация хорошая.

### Примеры
---
Наследование реализации:
Мы создаем базовый родительский класс вывода логов, который реализует метод send, который будет у всех потомков. И на основе базового родителя реализуются дети, которые не меняют класс родителя, а внедряют свое на основе имеющегося, без оглядки на домен. В данном случае реализуется потомок ZeroLog, который дополнительно реализует контроллер, который следит за переполнением буфера.
```python
class BaseLogger:
	def __init__(self, context, types, buffer):
		self.context = context
		self.types = types
		self.buffer = buffer
	
	def send(self, msg):
		if len(msg) > len(buffer):
			return
		self._flush(msg)

class ZeroLog(BaseLogger):
	def __init__(self, context, types, buffer):
		super().__init__(context, types, buffer)
		self._buffer_control = BufferController()
```

Наследование льготы:
В данном случае мы создаем базовый класс `response` на запросы, от этого класса наследуются различные виды ответов, которые реализуют свои формы "ответов".

class BaseApiReponse:

**Льготное наследование**: наследование класса FailedAmazonS3Response от класса HttpResponse. Ответ про сбой от хранилища Amazon S3 -- по сути, обычный ответ на http-запрос, который внутри себя содержит как стандарные сведения, так и дополнительную уточняющую информацию.

```java
class HttpResponse {
    private int statusCode;
    private byte[] body;

    public HttpResponse(int statusCode, byte[] body) {
        this.statusCode = statusCode;
        this.body = body;
    }
}

class FailedAmazonS3Response extends HttpResponse {
    private String key;
    private String bucketName;
    private String region;

    public FailedAmazonS3Response(String key, String bucketName, 
            String region, int statusCode, byte[] body) {
        super(statusCode, body);
        this.key = key;
        this.bucketName = bucketName;
        this.region = region;
    }
}
```

**Python**

**Наследование реализации:** Имеется класс Creature, и потребовалось добавить расу, которая накапливает статусы состояния, и не расходует их по мере течения игрового времени. При этом такой режим можно включать и выключать.  
Данный пример похож на наследование с расширением. Разница прежде всего в акцентах: в случае наследования с расширением мы вообще не задумываемся о реализации, а только расширяем набор операций. В случае наследования реализации по каким-то причинам решено было её учитывать, и в целом этого желательно всегда избегать, хотя на практике удаётся далеко не всегда.

```python
class Creature():
    def __init__(self, hp):
        self.hp = hp
        self.resistances = {}
        self.statuses = []

    def end_turn(self):
        for status in self.statuses.copy():
            status.tick(self)
            if status.duration <= 0:
                self.statuses.remove(status)

class AnotherCreature(Creature):
    def __init__(self, hp):
        super().__init__(hp)
        self.suppress = True

    def end_turn(self):
        if not self.suppress:
            super().end_turn
```

**Льготное наследование:** Класс-концовка игры. Мы создаем класс-событие, выполняющее нужные действия при завершении игры, и наследуемся от него, задавая новое описание. Например, если в игре потребуется создать тупиковую ветку, приводящую к концу игры, мы просто создадим наследника корневого завершения с новым описанием.

```python
class GameOver():
    same_text = ''

    def __init__(self):
        print(self.same_text)
        print('Game Over!')


class HappyEnding(GameOver):
    same_text = 'And they lived happily ever after'


class BadEnding(GameOver):
    same_text = 'And died on the same day. Today.'
```
