# Класс как тип

В данном уроке мы разберем другой подход, а именно будем рассматривать класс как тип!


### Отношение 
---

При рассмотрении класса как типа, мы пытаемся  совместить принцип повторного использования кода с расширяемостью, что не получалось с модулями. Для этого в типах применяется полиморфизм и динамическое связывание.

Рассмотрим отношение "класс как тип" с двух сторон:

**Отношение включения множеств** — множество всех объектов «Двусвязный список» является подмножеством множества всех объектов «Универсальный список». Причём эти два типа не обязательно наследуются напрямую — между ними может быть цепочка промежуточных типов.

**Отношение относительно операций** — «каждый двухсвязный список является универсальным списком» означает, что любая операция, применимая к родительскому универсальному списку, применима также и к дочернему двухсвязному списку, но не наоборот. Операции могут быть переопределены в наследнике.

То есть в случае "класс как тип" мы получаем не расширение родительского класса, а **специализацию**, что прямо противоположно подходу "класс как модуль".

### Парадокс расширения-специализации
---

Чем глубже мы идём по цепочке наследования вниз, тем к меньшему количеству объектов применим тип — мы движемся в направлении специализации. То есть специализация сужает применимость типа. Это и есть парадокс: расширяя иерархию вниз, мы фактически сужаем круг объектов, к которым применим каждый следующий тип.

### Важное правило про отношение «является»
---

Важно понимать, что:
	**Отношение "является" всегда выражает отношение только между категориями (типами, классами)**.

Так как нередко начинают сравнивать объекты с категориями, например:
"объект двусвязного списка list_1 является универсальным списком"

К этому важно понимать терминологию и не путать между собой ее:
	Термин «подкласс» некорректен, потому что смешивает два разных смысла: подмножество и наследование. Это создаёт путаницу.Правильные термины: наследник и родитель, потомок и предок, прямой/собственный потомок и предок.

### Динамическое связывание
---

**Динамическим связыванием** - это механизм при котором выбор итогового типа ложится на компилятор и происходит на этапе компиляции.

### Статическое связывание - плохо?
---

Почему статическое связывание это плохо?
Плохо по тому, что будет определяться итоговый тип по переменной, а не реальному объекту, который вложен. Это остро ощущается в C++, где методы по-умолчанию невиртуальные, поэтому при вот такой конструкции:

```c++
class Animal {
public:
    void sound() { cout << "звук"; }  // не virtual — статическое связывание
};

class Dog : public Animal {
public:
    void sound() { cout << "гав"; }
};

Animal* animal = new Dog();  // переменная типа Animal, объект типа Dog
animal->sound();             // выведет "звук", а не "гав"
                             // компилятор смотрит на тип переменной — Animal
                             // и на этапе компиляции уже решил вызвать Animal::sound()
```

Придется лезть в закрытый для изменений класс Animal и делать методы виртуальными.
В других популярных языках (Go, Python, Java) методы виртуальные, поэтому у них всегда идет именно динамическое связывание (если рассматриваем кейс про конкретное определение объекта через абстрактный тип).

### Наследование общий случай
---

В общем случае наследование **позволяет создавать новые классы, которые и расширяют, и специализируют уже существующие классы**, реализуется это с помощью трех механизмов:
- Переопределение методов — наследник перезаписывает логику метода родителя под свою специфику.
- Полиморфные объекты — переменная типа родителя может содержать объект любого наследника.
- Динамическое связывание — при вызове метода через полиморфную переменную выбирается реализация реального объекта, а не родителя.
Все три механизма работают вместе — убери любой из них и полноценного наследования не получится.

### Плоский класс и скрыте детали
---

Если класс в дальнейшем не собирается наследоваться, тогда он будет являться плоским, и необходимо скрыть все детали наследования, которые прослеживаются во внутренней реализации.
Достигается это с помощью:
- приватных методов и полей

Пример:

Плохо - наследование открыто, объект типа наследника, может спокойно вызвать метод родителя:

```python
class BaseLogger:
    def _format(self, message: str) -> str:
        return f"[LOG]: {message}"

class FileLogger(BaseLogger):  # пользователь видит наследование
    def __init__(self, path: str):
        self.path = path

    def log(self, message: str):
        formatted = self._format(message)  # метод родителя виден снаружи
        # пишем в файл


# пользователь класса видит всю цепочку
logger = FileLogger("app.log")
logger._format("test")  # может вызвать метод родителя напрямую — это плохо
```

Хорошо - внутренние детали скрыты:

```python
class BaseLogger:
    def __format(self, message: str) -> str:  # приватный метод
        return f"[LOG]: {message}"

class FileLogger(BaseLogger):
    def __init__(self, path: str):
        self.__path = path  # приватный атрибут

    def log(self, message: str):  # единственное что видит пользователь
        formatted = self.__format(message)
        # пишем в файл


# пользователь видит только чистый интерфейс
logger = FileLogger("app.log")
logger.log("сообщение")   # только это — больше ничего
```

Во всех остальных случаях наследование требуется ровно в тех границах, которые обеспечивают необходимую логику полиморфизма и динамического связывания в проекте.

### Пример
---

```go
type DBReader interface {
	ReadData() []byte
}

type MongoDB struct {
	conn *mongo.Conn
}

func (m *MongoDB) ReadData() []byte {
	// ...
}

type PostgresDB struct {
	conn *postgres.Conn
}

func (p *PostgresDB) ReadData() []byte {
	// ...
}

// динамическое связываение через интерфейс
func ReadInfoFromDifferentDB(r DBReader) []byte {
	return r.ReadData()
}

databases := []any{PostgresDB{}, MongoDB{}}

for _, db := range databases {
	// приведение типов
	
	// каждый из типов удовляетворяет интерфейсу
	// на этапе runtime происходит динамическое связывание
	// и выбирается тип влоэженного объекта 
	ReadInfoFromDifferentDB(db)
}
```
