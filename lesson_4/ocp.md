# Реализация расширяемости модулей. The Open Closed Principle (OCP)

В этом уроке мы поговорим о проблеме расширяемости модулей, а именно о реализации этой расширяемости.

Подходы следующие:

1. Расширить свой класс, наследуя другой и изменяя его настройки

Например, мы делаем свою административную панель с помощью фреймворка. Для того, чтобы представить в панели какой-нибудь объект (например форму для заполнения) мы используем классы, которые уже реализованы в нашем фреймворке для этого, только кастомизируем их по своему, вкладывая свои атрибуты и переопределяя методы. Этого можно достигнуть за счет описания интерфейса с помощью отложенных методов, то есть формально мы формируем порядок исполнения программы для определенного класса, который зафиксирован, и наследник обязан реализовать нужные методы, чтобы программа работала корректно., например:

```python
class DataProcessor:
    def run(self):
        data = self.load()      # отложенный — не реализован здесь
        result = self.process() # отложенный — не реализован здесь
        self.save(result)       # отложенный — не реализован здесь
```

далее ты наследуешь этот класс и дополняешь его своими настройками и переопределением методов, таким образом не ломается общая логика для работы класса (так как порядок запуска еобходимых методов определен), но ты его настраиваешь под себя!

2. Либо напрямую конфигурировать класс за счет, например, скриптов

Например, Django-приложение можно расширить, просто поменяв `settings.py`.

### Паттерны проектирования
---

Также в данном деле очень важно правильно использовать паттерны проектирования, не просто вставлять по шаблону, а на основе определенного паттерна строить собственный класс, дополняя его под свою задачу. То есть твоя программа не должна просто состоять из паттернов - "как по книжке", а тебе необходимо адаптировать те или иные паттерны под свой домен.

### The Open Closed Principle (OCP)
---

Данный принцип гласит о том, что:

- Модуль считается **открытым**, если его можно продолжать расширять
- Модуль считается **закрытым**, когда им можно только пользоваться

Для наглядности покажем иерархию классов, где показаны как закрытые так и открытые и есть обоснования к тому, почему они выбраны именно такими.


### Примеры (OCP)
---

1. Logger
Допустим нам необходимо сделать собственный логгер - обертку над существующим решением, для вывода различных сообщений, на различных уровнях, с разной информацией и метаданными. Пишем мы свой логгер, потому что нам не достает функционала в тех логгерах, которые есть в open-source commuinity. 
Тогда `slog.Logger` закрыт — мы не трогаем его код. `SlogHandler` открыт — он реализует интерфейс `slog.Handler`, и мы можем создавать новые обработчики, не меняя ни `slog.Logger`, ни уже написанные обработчики.:

```go
package logger

import (
	"context"
	"fmt"
	"log/slog"
	"runtime"
)

// SlogHandler реализует slog.Handler для интеграции с log/slog.
// Позволяет библиотекам, использующим slog, писать через наш логгер.

type SlogHandler struct {
	logger *Logger
	attrs  []slog.Attr
	group  string
}

```

2. Payment systems
Например, нам необходимо реализовать собственную платежную систему. Каждая из разных видов платежных систем реализована похожим образом, в данном случае базовый класс `Paymenter` - является закрытым классом, так как предоставляет свод методов необходимых для корректной работы платежной системы, а класс `PaymentSystemMIR` будет являться открытым классом, так как нам необходимо изменить методы привычной платежной системы.

```go
package paymenter

type Paymenter interface {
	Pay()
	GetPayStatus()
	Abort()
}
```

```go
package mir

type PaymentSystemMIR struct {
	// attrs
}

func (mir *PaymentSystemMIR) Pay() {}
func (mir *PaymentSystemMIR) GetPayStatus() {}
func (mir *PaymentSystemMIR) Abort() {}
```

```go
package service

import (
	"paymenter"
)

func ProccedPayment(paymenter Paymenter) {
	// ...some logic
}
```

Пример нарушения принципа:

```go
func ProccedPayment(system string) {
    if system == "MIR" {
        // логика MIR
    } else if system == "VISA" {
        // логика VISA
    }
}
```

Каждый раз, когда появляется новая платёжная система, ты лезешь в эту функцию и правишь её — это нарушение OCP. С интерфейсом ты просто создаёшь новый тип и передаёшь его, старый код не трогается.
### Итог
---

**Код должен быть открыт для расширения, но закрыт для модификации**

Вот основная мысль, которую важно запомнить!
Также необходимо понять, что для реализации данного принципа в ЯП пользуются либо наследованием, либо имплементацией строгого интерфейса.
